# Project Structure

dcrdata is organized as multiple Go modules in a single repository.

## Go Modules

| Path          | Module                                   | Purpose                                                                                        |
| ------------- | ---------------------------------------- | ---------------------------------------------------------------------------------------------- |
| `.`           | `github.com/decred/dcrdata/v8`           | Root module — shared packages (blockdata, mempool, pubsub, txhelpers, stakedb, rpcutils, etc.) |
| `cmd/dcrdata` | `github.com/decred/dcrdata/cmd/dcrdata`  | Main executable — block explorer + APIs                                                        |
| `db/dcrpg`    | `github.com/decred/dcrdata/db/dcrpg/v8`  | PostgreSQL backend                                                                             |
| `exchanges`   | `github.com/decred/dcrdata/exchanges/v3` | Exchange rate bot                                                                              |
| `gov`         | `github.com/decred/dcrdata/gov/v6`       | Governance (agendas, Politeia)                                                                 |

Dependent modules use `replace` directives in their `go.mod` to point at local paths.

## Root-level Packages

```
api/types/        Exported API types shared by dcrdata and Insight APIs
blockdata/        Primary block data collection and chain monitoring
db/
  cache/          Caching layer for dcrpg
  dbtypes/        Common DB data types
  dcrpg/          PostgreSQL backend (own module)
exchanges/        Exchange rate data collection (own module)
explorer/types/   Types used by explorer pages
gov/              Governance packages (own module)
mempool/          Mempool monitoring and data collection
netparams/        Network TCP port definitions
pubsub/           WebSocket pub-sub server
rpcutils/         dcrd RPC client helpers
semver/           Semantic versioning types
stakedb/          Ticket/stake tracking (Badger-backed)
txhelpers/        Block and transaction processing utilities
trylock/          Non-blocking mutex utility
```

## cmd/dcrdata — Main Application

```
cmd/dcrdata/
  main.go                   Entry point
  config.go                 CLI/env configuration
  internal/
    api/                    dcrdata REST API handlers and router
      insight/              Insight-compatible API
    explorer/               Block explorer page handlers, templates, websocket
    middleware/             HTTP middleware (logging, rate limiting, etc.)
    notification/           dcrd notification handling and sync orchestration
  public/
    index.js                Webpack entry point; registers all Stimulus controllers
    js/
      controllers/          Stimulus controllers (one per page feature, *_controller.js)
      helpers/              Shared JS utility functions
      services/             JS services (websocket, etc.)
    scss/                   SCSS source files; application.scss is the entry point
    dist/                   Webpack build output (do not edit manually)
    images/                 Static image assets
    fonts/                  Web fonts
  views/                    Go html/template files (.tmpl), one per page
    extras.tmpl             Shared template definitions (partials, helpers)
```

## Key Conventions

- Each Go package has a `log.go` file that initializes its named logger via `decred/slog`.
- Internal packages under `cmd/dcrdata/internal/` are not importable outside the module.
- Frontend JS follows the Stimulus pattern: one `*_controller.js` per page, registered in `public/index.js`.
- SCSS is split by concern (layout, typography, themes, per-page files); `application.scss` is the entry point.
- HTML templates in `views/` use Go's `html/template`; `extras.tmpl` contains shared template definitions.
- The `public/dist/` folder is generated by Webpack and should not be edited directly.
- Test files follow standard Go conventions (`_test.go` suffix, same package or `_test` package).
- Build tags `pgonline` and `chartdata` gate integration tests that require a live PostgreSQL instance.
- The `sample-dcrdata.conf` file documents all available configuration options.
